## 🏗️ AWS 솔루션스 아키텍처 개요 (High-Level)

AWS 아키텍처는 무수히 많은 조합이 가능하지만, 큰 틀에서는 **비용, 가용성, 내구성, 확장성**을 고려하여 최적의 컴포넌트를 선택하는 과정입니다.

### 1. 사용자 접점 및 전송 레이어 (Edge & DNS)

사용자가 애플리케이션에 접속할 때 가장 먼저 만나는 지점입니다.

* **DNS 레이어 (Route 53):** 사용자의 도메인 요청을 적절한 IP로 라우팅합니다.
* **CDN 레이어 (CloudFront):** 전 세계 Edge Location을 통해 정적/동적 콘텐츠를 캐싱하여 응답 속도를 높입니다.
* **S3/Glacier:** 정적 에셋(이미지, 동영상 등)의 원본 저장소 역할을 합니다.
* **Custom Origin:** S3 외에도 일반적인 웹 서버로부터 데이터를 가져올 수 있습니다.



### 2. 웹 및 엔트리 포인트 레이어 (Web Layer)

사용자의 요청을 받아 컴퓨팅 자원으로 전달하는 관문입니다.

* **로드 밸런싱:** CLB, **ALB**(L7), **NLB**(L4)를 통해 트래픽을 분산합니다.
* **API Gateway:** 서버리스 아키텍처나 API 관리 시 사용합니다.
* **Elastic IP (EIP):** 아주 단순한 구조에서 고정 IP가 필요할 때 사용하지만, 복잡한 아키텍처에서는 지양합니다.

### 3. 컴퓨팅 레이어 (Computing Layer)

실제 비즈니스 로직이 실행되는 곳입니다.

* **서버 기반:** EC2, Auto Scaling Group (ASG)
* **컨테이너:** ECS, Fargate
* **서버리스:** AWS Lambda
* **분석/배치:** EMR (빅데이터), Batch (배치 작업)

### 4. 데이터 저장 및 데이터베이스 레이어 (Storage & DB)

데이터의 성격에 따라 적절한 저장소를 선택합니다.

* **관계형(SQL):** RDS, Aurora
* **NoSQL:** DynamoDB (고성능 Key-Value)
* **검색/분석:** OpenSearch (ElasticSearch), Redshift (데이터 웨어하우스)
* **파일/블록 스토리지:** EBS (인스턴스 전용), EFS (공유 파일 시스템), Instance Store (임시 저장)

### 5. 캐싱 및 세션 관리 레이어 (Caching & Session)

성능 최적화와 상태 정보 유지를 위해 사용합니다.

* **ElastiCache:** Redis/Memcached를 통한 고성능 캐싱 (가장 추천되는 선택지)
* **DynamoDB (+DAX):** 세션 저장소로 활용 가능하며, DAX를 통해 가속화할 수 있습니다.
* **RDS:** 세션 저장이 가능하나, 성능/스케일 측면에서 ElastiCache보다 불리합니다.

### 6. 애플리케이션 통합 및 디커플링 (Integration)

마이크로서비스 아키텍처(MSA)에서 각 요소를 분리하고 유연하게 연결합니다.

* **메시징:** SQS (큐), SNS (알림/게시)
* **스트리밍:** Kinesis (실시간 데이터 분석)
* **하이브리드/마이그레이션:** Amazon MQ
* **워크플로우:** Step Functions (비즈니스 로직 오케스트레이션)

## 💡 솔루션스 아키텍트의 핵심 역량

시험과 실무에서 가장 중요한 것은 **"어떤 상황에서 어떤 조합이 최적인가?"**를 판단하는 것입니다.

| 고려 요소 | 설명 |
| --- | --- |
| **비용 (Cost)** | 성능을 유지하면서 가장 저렴한 조합인가? |
| **가용성 (Availability)** | 장애가 발생해도 서비스가 유지되는가? (Multi-AZ 등) |
| **내구성 (Durability)** | 데이터가 유실되지 않고 안전하게 보관되는가? (S3 등) |
| **확장성 (Scalability)** | 트래픽 증가에 따라 유연하게 대응할 수 있는가? |
