## 1. AWS Trusted Advisor 개요

Trusted Advisor는 사용자의 AWS 환경을 분석하여 **최적의 모범 사례(Best Practices)**를 권장하는 서비스입니다. 별도의 설치가 필요 없는 에이전트리스 서비스입니다.

### 6가지 핵심 카테고리

시험에서는 특히 **비용, 서비스 제한, 운영 우수성** 관련 내용이 자주 출제됩니다.

1. **비용 최적화 (Cost Optimization)**: 사용되지 않는 자원 식별.
2. **성능 (Performance)**: 처리량 및 효율성 개선.
3. **보안 (Security)**: 권한 설정 및 노출 위험 감지.
4. **장애 허용성 (Fault Tolerance)**: 고가용성 및 중복성 체크.
5. **서비스 제한 (Service Limits)**: 리소스 사용량이 한도에 도달했는지 확인.
6. **운영 우수성 (Operational Excellence)**: 운영 효율성 및 프로세스 점검.

## 2. 서포트 플랜별 기능 차이

Trusted Advisor의 모든 기능을 활용하려면 반드시 상위 서포트 플랜이 필요합니다.

| 구분 | Basic & Developer | Business & Enterprise |
| --- | --- | --- |
| **Trusted Advisor 범위** | **7개의 핵심 체크**만 제공 | **모든 체크 항목 (Full)** 제공 |
| **알림 (Notification)** | 이메일 알림 설정 가능 | 이메일 알림 + **CloudWatch 알람** |
| **프로그래밍 방식** | 액세스 불가 | **AWS Support API**를 통한 액세스 |

## 3. Trusted Advisor의 한계와 보완책

시험 문제에서 Trusted Advisor가 정답이 아닌 경우를 구분하는 것이 중요합니다.

* **S3 버킷 보안**:
  * Trusted Advisor는 버킷 자체의 **공개 여부**는 체크하지만, **버킷 내부의 개별 객체(Object)** 노출 여부는 확인하지 못합니다.
  * **해결책**: Amazon EventBridge, S3 Event Notifications, 또는 **AWS Config** 규칙을 사용하여 준수 사항을 확인해야 합니다.

* **서비스 제한(Quotas) 관리**:
  * 모니터링은 가능하지만, 제한 수치를 자동으로 늘려주지는 않습니다.
  * **해결책**: Support Center Case를 수동으로 열거나, **Service Quotas API**를 사용하여 프로그래밍 방식으로 요청해야 합니다.

## 4. 고급 아키텍처: 서비스 제한 자동 모니터링 및 대응

규모가 큰 엔터프라이즈 환경에서는 여러 계정의 서비스 제한을 중앙에서 관리하는 아키텍처가 권장됩니다.

### 아키텍처 흐름

1. **감지**: Lambda 함수가 주기적으로 Trusted Advisor를 새로고침하거나 Service Quotas(예: vCPU 제한)를 체크합니다.
2. **이벤트 발생**: 임계치에 도달하면 **Amazon EventBridge** 규칙이 실행됩니다.
3. **중앙 집중화**: 보조 계정의 이벤트를 **중앙 계정의 Event Bus**로 통합하여 관리합니다.
4. **대응 및 알림 (Targets)**:
   * **Lambda**: 서비스 제한 요약 보고서 생성 또는 **Service Quotas API를 호출하여 자동 한도 증액 요청**.
   * **SNS/SQS**: 관리자에게 즉각적인 이메일 알림 전송.
   * **Slack Notifier**: Slack 웹훅을 통해 실시간 알림 (자격 증명은 Parameter Store에 안전하게 보관).
   * **DynamoDB/DLQ**: 처리되지 못한 이벤트 저장 및 로깅.

* "서비스 제한을 자동으로 관리"하고 싶다면 **Service Quotas API + Lambda + EventBridge** 조합을 기억하세요.
