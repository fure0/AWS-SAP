## 1. Aurora Serverless

예측 불가능하거나 간헐적인 워크로드에 최적화된 구성입니다.

* **핵심 개념**: 실제 사용량에 따라 컴퓨팅 용량이 **자동 스케일링**되며, 사용하지 않을 때는 0으로 축소되어 비용을 절감합니다.
* **주요 특징**:
* **용량 계획 불필요**: 초 단위 과금 체계로 비용 효율성이 높음.
* **프록시 플릿(Proxy Fleet)**: 클라이언트는 Aurora가 관리하는 프록시를 통해 연결되며, 백엔드의 복잡한 스케일링 과정을 알 필요 없이 하나의 엔드포인트만 사용합니다.


* **Data API**:
* 전통적인 JDBC/ODBC 연결 대신 **보안 HTTP 엔드포인트**를 통해 SQL을 실행합니다.
* **Secrets Manager**와 연동되어 인증 정보를 관리하므로 애플리케이션에 자격 증명을 저장할 필요가 없습니다. (Lambda 환경에서 유용)

## 2. Aurora RDS Proxy & 읽기 전용 엔드포인트

애플리케이션과 데이터베이스 사이에서 연결 효율성을 극대화합니다.

* **연결 풀링**: 수천 개의 동시 연결을 관리하여 Lambda와 같은 서비스의 데이터베이스 부하를 줄입니다.
* **엔드포인트 분리**:
* **Primary 인스턴스**: 읽기/쓰기 수행.
* **Reader 전용 엔드포인트**: 읽기 전용 복제본(Read Replica)들에 대한 트래픽을 분산 처리.


* **보안**: Secrets Manager 및 IAM 정책을 통해 데이터베이스 액세스 권한을 엄격히 제어합니다.

## 3. Aurora Global Database

리전 간 복제를 통해 전 세계적인 성능 향상과 재해 복구(DR)를 제공합니다.

### 구성 및 성능

* **구조**: 1개의 기본(Primary) 리전 + 최대 **5개의 보조(Secondary) 리전**.
* **복제 속도**: 리전 간 복제 지연 시간 **1초 미만**.
* **확장성**: 각 보조 리전당 최대 16개의 읽기 전용 복제본 배치 가능.
* **복구 목표(DR)**: 리전 장애 시 다른 리전을 승격하는 데 **RTO 1분 미만** 소요.

### 쓰기 전달 (Write Forwarding)

* 보조 리전의 애플리케이션이 보조 리전 데이터베이스에 쓰기 요청을 보내면, 이를 **Primary 리전으로 자동 전달**하는 기능입니다.
* **장점**: 애플리케이션이 리전마다 다른 쓰기 엔드포인트를 관리할 필요가 없어 아키텍처가 단순해집니다.

## 4. RDS에서 Aurora로의 마이그레이션 전략

기존 RDS(MySQL/PostgreSQL)를 Aurora로 전환하는 두 가지 주요 방법입니다.

| 방법 | 프로세스 | 특징 |
| --- | --- | --- |
| **스냅샷 복원** | RDS 스냅샷 생성 → Aurora로 복원 | 가장 간단하지만, 복원 중 데이터 변경분 처리를 위해 다운타임이 필요할 수 있음. |
| **읽기 복제본 활용** | RDS의 **Aurora 읽기 복제본** 생성 → 동기화 → Aurora 승격 | 복제 지연이 거의 없을 때 승격하므로 **다운타임을 최소화**하며 마이그레이션 가능. |

---

### 💡 추가 팁 (PostgreSQL 관련)

PostgreSQL용 Aurora의 경우, **RPO(복구 시점 목표)** 설정을 통해 데이터 손실 가능성을 엄격하게 관리하고 지연 시간을 최적화할 수 있는 관리 기능이 포함되어 있습니다.
